## 一、项目概述

PolyMatrix Engine 是一款基于 **Python asyncio** 构建的**机构级 Polymarket 自动化做市商系统**。系统通过实时采集链上订单簿数据、监控仓位变化、自动执行双边报价策略，实现对 Polymarket 预测市场的全天候、低人力干预的流动性供给。

**核心技术栈：**
- **语言框架**：Python 3.11 + asyncio（高性能异步架构）
- **通信中间件**：Redis（Pub/Sub + Stateful KV，双重用途）
- **持久化层**：PostgreSQL（SQLAlchemy async）
- **订单执行**：Polymarket CLOB API（py-clob-client）
- **市场数据**：WebSocket（实时订单簿）+ REST API（引导 + 对账）
- **监控面板**：Streamlit（实时状态仪表盘）

## 三、核心技术亮点

### 3.1 内存优先的持仓状态管理

**问题背景：** 传统做法在每次行情tick时读取数据库获取持仓，在高频率报价场景下会造成数据库成为瓶颈。

**解决方案：** `InventoryStateManager` 实现内存优先架构：

```
读取路径：内存 → (空指针) → DB懒加载 → 回填内存
写入路径：内存立即更新 → 异步队列 → DB持久化
```

**关键实现：**
- **Singleton模式**：全局唯一状态实例，避免多实例不一致
- **有界队列**：持久化队列 `maxsize=1000`，防止DB故障时内存溢出
- **优雅关闭**：收到SIGTERM时等待队列排空，确保零数据丢失
- **时间戳保护**：写入前检查 `updated_at`，防止旧数据覆盖新数据

```python
# 核心状态结构（每市场）
{
    "yes_exposure": float,           # YES持仓股数
    "no_exposure": float,             # NO持仓股数
    "yes_capital_used": float,        # 买入YES消耗的USDC
    "no_capital_used": float,         # 买入NO消耗的USDC
    "pending_yes_buy_notional": float,# 待成交BUY订单名义值
    "realized_pnl": float,            # 已实现盈亏
    "last_local_fill_timestamp": float # 本地成交时间戳（对账保护）
}
```

### 3.2 统一公允价值定价模型

**核心创新：** YES引擎计算锚定FV并发布到Redis，NO引擎消费并推导 `FV_no = 1 - FV_yes`，确保双边报价始终协调一致。

```
FV_yes = clamp(mid_yes + OBI_yes * 0.015, 0.01, 0.99)
     mid_yes = (best_bid + best_ask) / 2
     OBI_yes = (best_bid_size - best_ask_size) / (best_bid_size + best_ask_size)
     动态价差 = base_spread * (1 + |OBI_yes|)
```

**业务意义：**
- OBI（订单不平衡指标）捕捉市场深度偏好
- 动态价差在流动性紧张时自动扩大，保护做市商
- NO侧无需独立计算，避免双边漂移

### 3.3 持仓状态机

系统为每个交易侧（YES/NO）维护独立状态机，根据持仓暴露量自动切换报价模式：

```
┌────────────────┐
│   QUOTING      │ ← 正常双向报价
└───────┬────────┘
        │ exposure >= 5
        ▼
┌────────────────┐
│ TWO_WAY_QUOTING│ ← 轻仓，继续双向但开启监控
└───────┬────────┘
        │ exposure >= liquidate_threshold (base*2)
        ▼
┌────────────────┐
│  LIQUIDATING   │ ← 开启maker平仓
└───────┬────────┘
        │ capital_used >= 90% * MAX
        ▼
┌────────────────────┐
│ EXTREME_LIQUIDATING│ ← Taker强平（跨价成交）
└───────┬────────────┘
        │
        ▼
┌────────────────┐
│  GRACEFUL_EXIT │ ← 外部信号（事件地平线/人工干预）
└───────┬────────┘
        │ exposure <= 1
        ▼
┌────────────────┐
│   TERMINATE    │
└────────────────┘
```

**跨Token锁定：** 当对手方持仓过大（>=50% * liquidate_threshold），本侧暂停BUY，防止双向被困。

### 3.4 差分报价（Diff Quoting）

**传统方案的缺陷：** 每tick取消所有订单再重新挂单 → 损失时间优先级 + 浪费手续费

**本系统方案：** 智能比对活跃订单与目标网格，三重保护避免频繁撤单：

| 保护机制 | 触发条件 | 效果 |
|----------|----------|------|
| 最小生命周期 | 订单年龄 < 8秒 | 防止刚挂即撤 |
| FV价格偏移 | 偏移量 < 0.005 | FV小幅波动不撤单 |
| 奖励spread范围 | 订单在奖励spread带内 | 留住奖励资格订单 |

```python
# Diff逻辑
保留：精确匹配（价格+数量）
取消：三重保护均不通过
创建：目标网格中无匹配项
```

### 3.5 定期硬重置（Ghost Order Prevention）

**问题：** 网络抖动、API超时可能导致订单在CLOB侧成交但本地状态未更新，形成"幽灵订单"锁定资金。

**v6.4解决方案：** 每5分钟执行一次硬重置：

```
1. 调用 CLOB cancel_all()（交易所侧全量取消）
2. 等待3秒（USDC保证金释放）
3. 本地OMS全量清理
4. REST API对账（强制模式，跳过时间保护）
5. 如对账失败：该tick冻结BUY订单
```

**防护效果：** 彻底杜绝幽灵订单导致的资金锁定。

### 3.6 自动对账机制

**双时基对账：** 系统同时依赖两个数据源：
- **WebSocket推送**：低延迟，但可能因网络问题丢失
- **REST API轮询**：可信来源，但有3-8秒延迟

**时间戳保护逻辑：**
```python
if (now - last_local_fill_timestamp) < 8秒:
    skip_overwrite  # 本地刚成交，不信任远端旧数据
else:
    overwrite_with_api_truth  # 以API数据为准
```

**比例调整：** 当API数据与本地不一致时，按比例调整capital_used，防止成本基准丢失。

### 4.2 Kill Switch 机制

当检测到风险敞口超标，系统执行多级熔断：

**单市场熔断：**
1. 写入数据库 `status = suspended`
2. 发布 Redis `control:{cid}` → `{"action": "suspend"}`
3. 报价引擎收到信号 → 立即取消全部门单 → 切换到 SUSPENDED 状态
4. OMS执行 `cancel_market_orders()`

**资金预检（Balance Precheck）：**
- 计算本市场可用预算 = MAX - (对手侧已用 + 其他市场已用)
- 按价格优先级保留BUY订单，丢弃尾部级别
- 确保新订单总名义值不超过可用预算

### 4.3 严格市值记账（Strict MTM Budgeting）

```python
held_inventory_value = yes_exposure * fv_yes + no_exposure * (1 - fv_yes)
pending_buy_notional = pending_yes_buy_notional + pending_no_buy_notional
strict_local_used = held_inventory_value + pending_buy_notional
```

**目的：** 即使 `capital_used` 因延迟未更新，基于当前市值的计算也能反映真实风险敞口，防止超买。

### 5.2 评分算法

**目标：** 在日奖励收益 × 流动性竞争度 × 时间衰减之间取得最优平衡。

```python
# V7.0 评分公式
daily_roi = reward_rate_per_day / rewards_min_size
base_score = daily_roi * reward_rate_per_day * (10000 / liquidity)

# 时间衰减（避免长尾市场）
if days_left > 180:  time_decay = 0.01   # 近到期超贵
elif days_left > 90:  time_decay = 0.5   # 中期惩罚
else:                 time_decay = 1.0   # 正常

final_score = base_score * time_decay
```

**直觉解读：**
- 高日奖励率 + 大奖励池 = 收益可观
- 低流动性（竞争少）= 更容易获得奖励
- 短期市场 = 时间衰减低，优先做市

### 5.3 硬过滤条件

| 条件 | 阈值 | 理由 |
|------|------|------|
| 奖励最小尺寸 | `rewards_min_size <= BASE_ORDER_SIZE` | 确保满足奖励资格 |
| 日奖励池 | `>= $50/天` | 过滤无价值市场 |
| 流动性 | `>= $20,000` | 只做有深度市场 |
| 事件地平线 | `> 24小时` | 避免临近结算风险 |
| 黑名单关键词 | 体育/博彩/临场投注 | 降低监管风险 |

## 七、系统局限性（诚实地讲）

我们相信坦诚是建立信任的基础。以下是当前系统的真实局限性：

### 7.1 性能瓶颈

| 问题 | 影响 | 当前缓解 |
|------|------|----------|
| Python GIL限制 | 多核CPU利用不足 | asyncio非阻塞设计缓解 |
| Redis单点 | 故障时系统停摆 | 后续可做Redis Cluster |
| 对账延迟 | 最多8秒的信任窗口 | 仅影响极端边界情况 |

### 7.2 功能局限

| 局限 | 说明 |
|------|------|
| **仅支持Binary YES/NO** | 尚不支持多结果市场 |
| **单一做市策略** | 固定网格，暂无 Viking/其他高级策略 |
| **被动平仓** | 强制平仓使用Taker（跨价），存在滑点 |
| **无对冲机制** | 无法跨市场对冲，依赖仓位管理 |
| **时区依赖** | 强依赖系统TZ设置，多实例部署需注意 |

### 7.3 订单执行局限

| 问题 | 说明 |
|------|------|
| **CLOB依赖** | 完全依赖Polymarket CLOB可用性 |
| **无价格保护** | 快速行情下可能以不利价格成交 |
| **最小订单尺寸** | Polymarket要求>=5股，可能限制灵活性 |

### 7.4 风险管理局限

| 局限 | 说明 |
|------|------|
| **无动态止盈** | 依赖状态机切换，无预设止盈逻辑 |
| **板块限制静态** | `MAX_EXPOSURE_PER_SECTOR`需手动调参 |
| **无机器学习风控** | 当前基于规则，风控模型待引入 |

### 7.5 AutoRouter局限

| 问题 | 说明 |
|------|------|
| **扫描间隔固定** | 默认1小时，可能错过短期机会 |
| **黑名单不完整** | 体育/博彩过滤可能误伤 |
| **无外部数据融合** | 暂不支持新闻/舆论数据辅助决策 |

---

## 八、技术架构总结

```
PolyMatrix Engine = 事件驱动架构 + 内存优先状态 + 多层风控 + 智能组合管理
```

**核心差异化优势：**

1. **生产级可靠性**：CircuitBreaker、Hard Reset、时间戳对账 — 面向极端行情设计
2. **零数据库热路径**：内存状态确保tick循环<10ms延迟
3. **统一定价Oracle**：YES/NO双边协调报价，杜绝价差漂移
4. **Ghost Order防护**：定期CLOB全量取消 + REST对账
5. **自动化组合管理**：雷达扫描 + 评分排序 + 板块限制

**适用场景：**
- Polymarket Binary市场做市
- 日奖励收益套利
- 预测市场流动性供给

**不适合场景：**
- 高频套利（延迟敏感）
- 多结果市场（暂不支持）
- 跨市场对冲（无此功能）

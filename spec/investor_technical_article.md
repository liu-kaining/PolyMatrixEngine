# PolyMatrix Engine：面向 Polymarket 的流动性基础设施，而非「又一个交易脚本」

**一句话**：在预测市场成交量指数级放大的背景下，我们将 **CLOB 做市**抽象为可长期运行的 **异步执行与风控系统**——**热路径零数据库、内存单一真相源、差分报价保队列优先权、四层风控与强制对账闭环**，并可选 **组合级 Auto-Router** 做赛道与事件视界约束下的自动选场。

面向投资人与战略合作伙伴的技术叙事（非实现级白皮书）。实现细节见仓库 `README.md` 与 `spec/architecture_diagrams/`（含 Mermaid 架构图）。

文中市场数据来自公开报道或第三方统计，**口径不一，仅用于说明趋势**；不构成业绩承诺。

## 一、为什么是现在：市场在变「深」，对「基础设施」提出要求

**2025–2026 年**，预测市场已从边缘实验走向**高成交量、高关注度、机构资金公开布局**的阶段：全平台年成交量在多家行业复盘里已达**约 500 亿美元量级**，头部平台高度集中；Polymarket 单平台在常见统计口径下多在**约 210–230 亿美元/年**区间。与此同时，**ICE（纽交所母公司）对 Polymarket 最高约 20 亿美元（$2B）量级的战略投资**、**约 80 亿美元（$8B）投前估值**等公开信息（路透社 2025-10），表明**传统交易所与清算基础设施**已开始与事件合约赛道发生制度化连接。

Polymarket 官方将 **做市商（MM）** 定义为持续挂限价单、**赚取价差**并改善流动性的角色，并配套 **Liquidity Rewards** 与部分市场的 **Maker Rebates**。这意味着：**谁能在低延迟、低风险、可运维的前提下规模化挂单，谁就更可能稳定吃到激励与价差**。

**真正的瓶颈**不在于「会不会调 API」，而在于——

- **状态一致性**：成交、撤单、断线、幽灵单与本地账本漂移，能否在秒级到分钟级内自愈？
- **资金效率**：预算是否按 **MTM + 未成交买单名义** 等**严格口径**计算，避免重复占用或越界？
- **执行质量**：是「全撤全挂」毁掉队列位置，还是 **差分** 保留时间优先权并降低限频风险？
- **组合维度**：多市场并行时，如何避免**单一赛道押注**与**临近结算事件风险**？

**PolyMatrix Engine** 正是围绕上述四类问题做的**工程化答案**。

## 二、架构总览：四条数据流、三个平面、一个热路径原则

系统不是单文件脚本，而是**可拆可合的分层架构**（与 `spec/architecture_diagrams/15_system_overview.md` 一致）：

| 平面 | 核心职责 | 代表模块 |
|------|----------|----------|
| **数据面** | 订单簿实时性、用户成交/撤单事件可靠入站 | Market WS + Gateway（本地簿合并、静默断线检测）；User WS（成交写账、驱动库存） |
| **状态面** | 热路径上的**单一真相源**；冷数据异步落库 | `InventoryStateManager`：内存快照；**有界**持久化队列；关停排空 |
| **执行面** | 与 CLOB 交互的韧性（熔断、超时、幂等友好） | OMS + `py-clob-client`；可选 Builder 归因 |
| **风控面** | 与报价路径**解耦**的持续监测与真相拉取 | Watchdog（敞口、对账、Kill Switch）；硬重置后与 Data API **强制对齐** |

**热路径铁律**：`QuotingEngine` 的 tick 内 **只读内存库存快照**（`get_snapshot`），**不读 Postgres**。成交先改内存，再进**异步队列**写库——这是 Python 系量化系统里最常见的性能坟场，被我们**从架构上剔除**。

**控制面**：FastAPI 提供生命周期与运维 API；Streamlit 提供驾驶舱（筛选、启停、敞口、日志）。**Auto-Router** 作为可选后台任务，与引擎**解耦**，只通过受控的「启动/退出市场」接口协作。

## 三、投资人应关注的六大差异化（也是我们愿意被尽调的重点）

### 1. 统一定价 Oracle：YES 锚定、NO 派生，双引擎不「各算各的」

每个二元市场部署 **YES / NO 两套 QuotingEngine**，但 **公平价值（FV）来自统一锚定**：以 YES 侧订单簿与模型得到锚定价与动态价差，NO 侧在一致约束下派生。好处是**同一事件上两侧报价逻辑同源**，降低「自我对冲失败」与盘口不一致带来的无谓磨损。

### 2. 差分报价（Diff Quoting）：保队列、省 API、降限频风险

与「每次重算就 cancel-all 再全量下单」不同，引擎按 **(方向, 价格, 规模)** 签名比对**目标网格**与**当前挂单**：**仅撤销偏离项、仅创建缺口**。这对做市的真实含义是：**尽量保留在 CLOB 队列中的时间优先权**，并显著减少撤挂次数——在官方速率限制下，这是**可持续运行**的关键。

### 3. 严格预算口径：MTM 持仓 + 双侧 Pending BUY，不是「拍脑袋仓位上限」

单笔与网格层的 BUY 规模受 **按市值（MTM）计价的库存** 与 **YES/NO 两侧未完成买单名义** 共同约束，并与 **`MAX_EXPOSURE_PER_MARKET`**、**`GLOBAL_MAX_BUDGET`** 联动；下单前 **Balance Precheck** 可自动**缩单或丢层**。这是对**资本效率与爆仓式超挂**的双重回答——也是机构风控问的第一组问题。

### 4. Maker 纪律：穿价保护，避免「赚激励反付 Taker」

价格被约束在 **不横穿订单簿** 的区间内（BUY 不高于可成交卖、SELL 不低于可成交买，含 tick 缓冲），避免在极端行情下**误成 Taker**、吞噬价差与激励逻辑。

### 5. 四层风控 + 「真相强制拉取」：不是「挂了风控两个字」

1. **报价前**：上述预算与预检。  
2. **Watchdog**：高频率审视 **实际已占用资本（capital_used）** 等硬指标，触发 **Kill Switch**（库表停牌 + Redis 控制信道 + 撤单）。  
3. **周期对账 + 时间窗**：与 Polymarket **Data API** 仓位对齐；对 **刚发生本地成交** 的时间窗内**禁止**用陈旧 REST 覆盖——避免「对账反成破坏者」。  
4. **周期性硬重置**：在受控流程下与交易所 **cancel_all** 路径协同（含超时与睡眠释放资金锁定），随后 **本地强制清障 + `reconcile_single_market(force=True)`**；若**强制对账失败**，当 tick **冻结新增 BUY**（保守面对不确定库存）。  

此外，**User WebSocket 重连**后可触发 **强制全量对账**，修补断线窗口内可能丢失的成交感知——在「拉长 REST 对账间隔、主依赖 WS」的运维策略下，这是**必要的安全阀**。

### 6. Auto-Router：官方激励雷达 + Gamma 元数据 + 组合约束

可选 **Auto-Router** 从 **CLOB 官方 rewards 目录**分页拉取市场，经黑名单与基础规模过滤后**多因子评分**（收益、竞争惩罚、事件视界等），再叠加 **单赛道槽位/敞口上限**、**临近结算驱逐**、**掉出 TopN 时的最小持有期（定力锁）** 等规则，自动维护 **Top-N 组合**。这是把「选市场」从人肉表格升级为**可配置策略模块**，与执行引擎**清晰边界**——利于未来接更多 **Alpha 信号源**而不污染核心执行层。

## 四、工程可信度：为什么团队「像认真做过生产系统的人」

- **并发与生命周期**：`EngineSupervisor` 保证单市场单侧**唯一任务**，退出时**干净回收**，降低幽灵进程与重复挂单。  
- **数据网关韧性**：Market 侧含**静默断线检测**与重连逻辑，避免「假连接真僵尸」。  
- **持久化不拖死主循环**：有界队列 + 关停排空，避免内存无限胀与悄悄丢单。  
- **OMS 熔断**：区分瞬态与非瞬态错误，避免无意义重试放大事故面。  
- **配置驱动**：关键风险参数 **环境变量化**，便于分环境审计与合规沟通。

## 五、价值定位与合作边界

以下仅陈述我们对赛道与产品能力的判断，**不构成任何收益或业绩承诺**（完整免责见文末）。

- **赛道**：预测市场流动性需求与官方做市激励并存，**专业化做市产能**仍稀缺。  
- **产品形态**：PolyMatrix Engine 是 **可部署、可运维、可审计** 的 **执行 + 风控 + 可选组合** 栈，而非演示级脚本。  
- **扩展叙事**：统一执行层之上，可叠加 **外部信号、自定义选场、多账户路由** 等——**执行与风控的厚度**是承接更大资金与更复杂策略的前提。

我们期待与认同 **「基础设施优先于单次押注」** 的资本与产业方深入交流。
